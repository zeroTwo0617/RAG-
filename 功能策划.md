# RAG 知识库问答系统 · 功能策划书

> 技术栈：Vue 3 + Vite + TypeScript + Pinia + Element Plus（前端）｜ Spring Boot + MyBatis-Plus + PGVector（后端）｜ PostgreSQL（数据 + 向量）
> 项目场景：**个人技术笔记问答** —— 上传自己的 Vue / Java / SQL 学习笔记，向 AI 提问，AI 基于笔记内容精准回答并标注引用来源
> 目标定位：简历项目，侧重展示「RAG 工程深度 + Vue/Java/SQL 全栈能力 + 质量保障意识」

---

## 一、项目概述

### 1.1 项目定位

一个基于 RAG（检索增强生成）的知识库问答系统。区别于直接调 LLM 的「套壳聊天」，本项目的核心壁垒是：

- 答案**只来自用户自己的文档**，不瞎编（faithfulness）
- 每个回答都**标注引用来源**（哪篇文档、哪一段），可追溯、可验证
- 检索链路**工程化可评测**（召回率、忠实度指标量化）

### 1.2 目标用户与场景

| 项 | 内容 |
|----|------|
| 知识库主体 | 用户本人的技术笔记（Markdown 为主，10-20 篇） |
| 使用者 | 用户自己 / 面试演示时的面试官 |
| 典型问题 | "Vue3 的 ref 和 reactive 区别是什么？" "MyBatis 怎么分页？" |
| 非目标 | 不接实时互联网、不做多轮复杂 Agent 编排（MVP 聚焦问答） |

### 1.3 技术栈总览

| 层 | 选型 | 说明 |
|----|------|------|
| 前端框架 | Vue 3（Composition API + `<script setup>`） | 组合式 API，逻辑复用强 |
| 构建工具 | Vite | 极速热更新 |
| 语言 | TypeScript | 类型安全，加分项 |
| 状态管理 | Pinia | 购物车式轻量状态（对话、用户态） |
| UI 库 | Element Plus | 中后台组件全，开发快 |
| HTTP | Axios（拦截器统一 token/错误） | 工程规范 |
| 后端框架 | Spring Boot 3 | 自动配置 + 内嵌 Tomcat |
| 安全 | Spring Security + JWT | 登录 + 后台权限 |
| 数据访问 | MyBatis-Plus | SQL 握在手里，省 CRUD 样板 |
| 向量库 | PostgreSQL + PGVector 插件 | 结构化数据 + 向量同库，复用你的 SQL 能力 |
| LLM / Embedding | 国内 API（DeepSeek / 通义 / 智谱，可配） | 测试期低成本 |
| 流式输出 | SSE（Server-Sent Events） | 打字机效果 |
| 部署 | 前端 Vercel / 后端 Docker + PostgreSQL | 可访问链接直接贴简历 |

### 1.4 简历价值（一句话）

> 独立从 0 搭建 RAG 知识库问答系统（Vue3 + Spring Boot + PGVector），实现基于结构的分块策略、混合检索 + Rerank 与流式引用溯源，含评测集与召回/忠实度指标，已部署可访问。

一句话覆盖：**全栈栈 + RAG 深度 + 质量保障 + 上线状态**。

## 四、数据库设计（PGVector）

### 4.1 表结构

**document（文档元数据）**

```sql
CREATE TABLE document (
    id          BIGSERIAL PRIMARY KEY,
    title       VARCHAR(255) NOT NULL,
    source_type VARCHAR(32)  NOT NULL,   -- markdown / pdf / url
    status      VARCHAR(32)  NOT NULL,   -- pending / parsing / indexed / failed
    owner_id    BIGINT       NOT NULL,
    chunk_count INTEGER      DEFAULT 0,
    created_at  TIMESTAMP    DEFAULT NOW()
);
```

**chunk（文本块 + 向量，核心表）**

```sql
CREATE TABLE chunk (
    id           BIGSERIAL PRIMARY KEY,
    document_id  BIGINT NOT NULL REFERENCES document(id) ON DELETE CASCADE,
    content      TEXT NOT NULL,                 -- 块文本内容
    section      VARCHAR(255),                  -- 所属章节（来自 Markdown 标题）
    chunk_index  INTEGER,                       -- 在文档中的序号
    meta_json    JSONB,                         -- 扩展元数据 {page, category, ...}
    embedding    vector(1536),                  -- Embedding 向量（维度随模型定）
    created_at   TIMESTAMP DEFAULT NOW()
);
-- 向量索引（IVFFlat，按召回速度取舍；也可用 HNSW）
CREATE INDEX idx_chunk_embedding ON chunk USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
-- 过滤索引
CREATE INDEX idx_chunk_doc ON chunk(document_id);
```

> 维度 `1536` 对应 OpenAI text-embedding-3-small / 通义 bge 等；选型确定后改维度即可。

**qa_log（问答记录，用于反馈与评测）**

```sql
CREATE TABLE qa_log (
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT,
    question    TEXT NOT NULL,
    answer      TEXT,
    sources     JSONB,            -- 引用的 chunk_id 列表 + 片段
    latency_ms  INTEGER,
    model       VARCHAR(64),
    created_at  TIMESTAMP DEFAULT NOW()
);
```

**feedback（用户反馈）**

```sql
CREATE TABLE feedback (
    id          BIGSERIAL PRIMARY KEY,
    qa_id       BIGINT REFERENCES qa_log(id),
    rating      SMALLINT,         -- 1 赞 / -1 踩
    comment     TEXT,
    created_at  TIMESTAMP DEFAULT NOW()
);
```

### 4.2 检索 SQL 示例（混合检索 + 元数据过滤）

```sql
-- 向量召回 Top-K（余弦相似度，取反即距离）
SELECT id, content, section, 1 - (embedding <=> :queryVec) AS score
FROM chunk
WHERE document_id IN (:allowedDocs)        -- 权限/范围过滤
ORDER BY embedding <=> :queryVec
LIMIT :k;

-- 关键词召回（ILIKE 或全文检索 tsvector，取交集后融合）
SELECT id, content, section,
       ts_rank(to_tsvector('simple', content), plainto_tsquery('simple', :kw)) AS score
FROM chunk
WHERE to_tsvector('simple', content) @@ plainto_tsquery('simple', :kw)
LIMIT :k;
```

> 融合策略：向量召回 Top-K 与关键词召回 Top-K 各取，按 score 归一化后加权合并（RRF 倒数排名融合是常见做法），再送 Rerank。

### 4.3 Flyway 迁移

`db/migration/V1__init.sql`（建表 + 建扩展 `CREATE EXTENSION IF NOT EXISTS vector;`）→ `V2__seed.sql`（插入 2-3 篇示例笔记 + 分块，便于本地一键看效果）。

### 4.4 种子数据

准备 3 篇示例 Markdown 笔记（Vue3 基础、Spring Boot 入门、SQL 优化），随迁移脚本预切分块入库，保证 `npm run dev` + 启动后端即可直接提问演示。

## 五、RAG 管线设计（核心）

### 5.1 离线入库管线（上传触发）

```
Markdown 文件
  → 解析（按 # 标题构建章节树，提取纯文本）
  → 分块（见 5.3 策略）
  → 标注元数据（source / section / chunk_index）
  → 调 Embedding API 生成向量
  → 批量写入 chunk 表（document.status = indexed）
```

### 5.2 在线问答管线（提问触发）

```
用户问题
  → Query 改写（可选：口语化 → 检索友好关键词，用 LLM 轻量改写）
  → 向量召回 Top-K（PGVector 余弦）
  → 关键词召回 Top-K（全文检索）
  → 融合（RRF 倒数排名融合）
  → Rerank（交叉编码器精排，取 Top-N）
  → 拼 Prompt（系统指令：只基于上下文、必须引用来源）
  → LLM 流式生成（SSE 逐字返回）
  → 前端展示答案 + 引用卡（点击展开原文片段）
```

### 5.3 分块策略（简历亮点，面试必讲）

采用**组合策略**，写入 README 说明取舍：

1. **第一层：按结构切** —— 以 Markdown 二级标题（`##`）为边界，每个标题下内容为一个初始块，保留语义完整
2. **第二层：长度二次切** —— 初始块超过 `MAX=400 字` 时，按句号/段落递归切到 ≤400 字
3. **重叠窗口** —— 相邻块重叠 `OVERLAP=80 字`，避免跨边界知识点被切断
4. **元数据** —— 每块记 `section`（所属标题路径）、`chunk_index`

> 经验值（中文场景甜点区）：块 300-500 字、重叠 50-100 字。但**必须用你的数据实测调优**，这不是死规矩——这点在评测章节闭环。

### 5.4 混合检索

- **向量召回**：语义相似（"怎么优化 SQL" 能命中 "索引使用建议"）
- **关键词召回**：精确匹配（专有名词、报错码、API 名必须命中）
- 单一召回各有盲区，融合后召回率显著提升（评测可量化对比）

### 5.5 Rerank

向量召回 Top-20 + 关键词 Top-20 → 融合得 ~30 候选 → 用轻量交叉编码器（如 `bge-reranker`）对 `(query, chunk)` 打分 → 取 Top-5 进 Prompt。Rerank 是效果提升的关键一环，且工程上只是多一次打分调用。

### 5.6 流式 SSE 输出

后端 `/api/qa/ask` 返回 `text/event-stream`，每生成一个 token 推一条 `data:` 事件；前端用 `EventSource` 或 Axios `onDownloadProgress` 累积渲染，实现打字机效果。引用来源在生成结束后单独推送一条结构化事件。

### 5.7 引用溯源

生成时要求 LLM 在答案中输出形如 `[1][2]` 的引用标记，后端映射回 `qa_log.sources`（chunk_id + 片段）。前端渲染为可点击卡片，点击展开原文。这是 RAG 区别于普通聊天的**硬指标**。

## 七、评测方案（质量保障，强差异化）

### 7.1 评测集构建

人工准备 30-50 条 `(问题, 期望答案要点, 应命中文档)` 三元组，覆盖：
- 单文档事实题（"Vue3 ref 是什么"）
- 跨文档综合题
- 拒答题（问笔记里没有的内容，期望回答"知识库未涵盖"）

### 7.2 核心指标

| 指标 | 含义 | 计算 |
|------|------|------|
| 召回率 Recall@K | 应命中文档是否进 Top-K | 命中数 / 总数 |
| 忠实度 Faithfulness | 答案是否全基于上下文、无编造 | LLM-as-judge 或人工 |
| 答案相关性 | 是否切题 | LLM-as-judge |
| 延迟 | 首字 / 总耗时 | 埋点统计 |

### 7.3 调优旋钮（闭环）

| 旋钮 | 影响 | 实验方式 |
|------|------|---------|
| 块大小 / 重叠 | 召回质量 | 跑评测集对比 |
| Top-K / Rerank 开关 | 精度 vs 成本 | A/B |
| Prompt 模板 | 忠实度 | 多版本对比 |
| 混合检索权重 | 召回率 | 调 RRF 权重 |

> 简历写法：「构建 40 条评测集，对比分块策略前后 Recall@5 从 X 提升到 Y，验证分块对召回的影响」——这句话直接证明你是真懂 RAG，不是只会调 API。
